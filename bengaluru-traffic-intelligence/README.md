# Bengaluru Traffic Intelligence Module
### Flipkart Grid Phase 2 | PS1: Poor Visibility on Parking-Induced Congestion

> **AI-driven parking intelligence** that detects illegal parking hotspots, quantifies
> their downstream impact on traffic flow, and enables Bengaluru Traffic Police (BTP)
> to shift from reactive patrol to predictive, targeted enforcement.

---

## Problem Statement

On-street illegal parking near commercial areas, metro stations, and events chokes
Bengaluru's carriageways. Today's enforcement is entirely patrol-based and reactive.
There is no heatmap of where parking violations most impact flow, and no data-driven
way to prioritise enforcement zones.

**Our solution:** A read-only intelligence plugin on top of BTP's ASTraM system that provides:
1. **Detect** — NLP classifier (TF-IDF + Logistic Regression) on `event_cause` + `description` text
2. **Quantify** — Cascade analysis: "tanker obstruction on ORR triggers 2.3 downstream incidents"
3. **Act** — Enforcement schedule: junction × day × time_period with officer deployment recommendations

## Quickstart

### 1 — Place your dataset

```bash
cp /path/to/your/dataset.csv backend/data/raw/incidents.csv
```

The dataset should have these key columns (from the BTP/ASTraM export):
`id, event_type, latitude, longitude, event_cause, requires_road_closure,
start_datetime, resolved_datetime, veh_type, age_of_truck, cargo_material,
corridor, junction, zone, police_station, priority, description, reason_breakdown`

### 2 — Backend setup

```bash
cd backend

# Install dependencies (Python 3.10+ recommended)
pip install -r requirements.txt

# Run full pipeline (~8–15 minutes for 3 lakh rows)
python pipeline/run_all.py
```

To run individual steps only:
```bash
python pipeline/run_all.py --only 01 02   # only steps 1 and 2
python pipeline/run_all.py --skip 05      # skip LightGBM training
```

### 3 — Start API server

```bash
# From backend/ directory
uvicorn api.main:app --reload --port 8000

# API docs: http://localhost:8000/docs
# Health check: http://localhost:8000/health
```

### 4 — Frontend

```bash
cd frontend
npm install
npm run dev

# Dashboard: http://localhost:5173
```

## API Endpoints

| Endpoint | Method | Description |
|---|---|---|
| `GET /health` | GET | Service health + data/model status |
| `GET /summary` | GET | KPI cards: incident counts, avg clearance, top corridor |
| `GET /incidents` | GET | Filtered incident list (params: zone, event_type, priority, parking_only, limit) |
| `GET /hotspots?hour=8&day=0` | GET | H3 GeoJSON weighted for a time slot (for map time slider) |
| `POST /predict` | POST | LightGBM clearance time prediction + SHAP values |
| `GET /risk-scores` | GET | Top 20 junctions by composite risk score |
| `GET /cascade` | GET | Cascade impact scores by junction and event type |
| `GET /enforcement-schedule` | GET | Officer deployment schedule by junction |
| `GET /validation` | GET | Model CV metrics, MAE, feature importance |
| `GET /policy-recommendations` | GET | Claude API plain-English enforcement recommendations |

## Pipeline Steps

| Step | Script | What it does | Output |
|---|---|---|---|
| 1 | `01_clean.py` | Load CSV, spatial filter, compute clearance_minutes, engineer 15+ features | `cleaned_incidents.parquet` |
| 2 | `02_parking_classifier.py` | TF-IDF + Logistic Regression classifier on text fields | `parking_classified.parquet` + `parking_classifier.pkl` |
| 3 | `03_proximity_scoring.py` | Vectorised Haversine to 30+ known zones; composite parking score | `proximity_scored.parquet` |
| 4 | `04_hotspot_analysis.py` | H3 hex surface (7×24 probability grid) + DBSCAN clustering | `hotspot_hexes.geojson` + `h3_surface.parquet` |
| 5 | `05_lgbm_model.py` | LightGBM + XGBoost ensemble, rolling temporal CV, SHAP | `lgbm_model.pkl` + `validation_results.json` |
| 6 | `06_cascade_analysis.py` | Time-window self-join on corridor to find downstream incidents | `cascade_scores.json` |
| 7 | `07_risk_scoring.py` | Weighted risk formula: frequency + clearance + closure + cascade | `risk_scores.json` |
| 8 | `08_enforcement_schedule.py` | Junction × day × time_period incident rate + deployment rec | `enforcement_schedule.json` |

## Research Backing

| Method | Research Basis |
|---|---|
| NLP classifier (TF-IDF + LR) | Springer 2025: NLP achieves high accuracy on traffic incident classification with noisy, manually-entered text |
| LightGBM regressor | Istanbul 2024–26 study (38,430 incidents): LightGBM best MAE vs RF and XGBoost |
| Ensemble (LGB + XGB) | Sydney arXiv 2406.18861: ensemble outperforms all individual models for incident duration |
| Rolling temporal CV | Standard practice for time-series — prevents data leakage from future into training |
| DBSCAN clustering | Validated for road-network-constrained clustering; Dijkstra-DBSCAN literature |
| H3 hexagonal grid | Used in production by Swiggy/Zomato for Bengaluru delivery zone analysis |
| Proximity POI scoring | Annals of GIS 2019: POI categories positively associated with parking violation density |
| Cascade analysis | Time-window self-join — transparent, auditable, no additional model uncertainty |

## MapMyIndia Integration

The frontend currently uses a canvas-based map placeholder. To integrate the
real MapMyIndia Mappls SDK:

1. Get a free developer API key at https://maps.mappls.com/
2. Add to `frontend/index.html`:
   ```html
   <script src="https://apis.mappls.com/advancedmaps/api/{YOUR_KEY}/map_sdk?layer=vector&v=3.0"></script>
   ```
3. Replace the `MapPlaceholder` component in `Tab1_SmartMap.jsx` with:
   ```js
   mapInstance.current = new mappls.Map('map-container', { center: [12.9716, 77.5946], zoom: 11 })
   ```
4. Add the H3 GeoJSON overlay from `GET /hotspots` as a map layer.

## Claude API (Policy Recommendations)

The `/policy-recommendations` endpoint uses `claude-sonnet-4-6` to convert
structured junction risk data into plain-English enforcement recommendations.

Set your API key before starting the server:
```bash
export ANTHROPIC_API_KEY=sk-ant-...
uvicorn api.main:app --reload --port 8000
```

## What This Is NOT

- Not a replacement for ASTraM — it reads from ASTraM data, never writes
- Not a real-time sensor system — it uses historical incident data with live simulation
- Not a black-box system — every risk score formula and SHAP value is auditable
